Урок 13: Резервное копирование и миграция в Oracle

Введение

Резервное копирование и восстановление — одна из важнейших обязанностей администратора баз данных. Как отмечается в официальной документации Oracle, "без адекватных резервных копий базы данных не может быть восстановления этой базы данных" . В этом уроке мы рассмотрим основные методы резервного копирования, инструменты миграции данных и вспомогательные средства оптимизации.

1. Физические и логические резервные копии

В Oracle существует два принципиально разных подхода к резервному копированию :

1.1 Физические резервные копии

Определение: Физическая резервная копия — это копия физических файлов базы данных: datafile, control file, archived redo logs .

Особенности:

Являются основой любой стратегии backup/recovery
Позволяют восстановить базу данных на момент сбоя (point-in-time recovery)
Работают только на уровне всей базы данных или отдельных tablespace
Виды физических бэкапов:

Тип	Описание	Режим работы	Требования
Offline (Cold) Backup	Копирование файлов при остановленной БД	DB закрыта	Можно делать в NOARCHIVELOG mode
Online (Hot) Backup	Копирование файлов при работающей БД	DB открыта	Требуется ARCHIVELOG mode
Online Backup (горячее копирование) выполняется по табличным пространствам :

sql
-- Перевод tablespace в режим резервного копирования
ALTER TABLESPACE users BEGIN BACKUP;
-- Копирование файлов (через OS команды)
! cp /u01/oradata/users01.dbf /backup/
-- Завершение режима
ALTER TABLESPACE users END BACKUP;
1.2 Логические резервные копии

Определение: Логическая резервная копия содержит логические данные — таблицы, хранимые процедуры, представления, экспортированные в бинарный файл .

Особенности :

Позволяют выборочно экспортировать/импортировать объекты (таблицы, схемы)
Независимы от физической структуры
Не поддерживают roll-forward восстановление
Представляют "снимок" данных на момент экспорта
Основные инструменты:

exp/imp — классические утилиты (устаревшие)
expdp/impdp — Data Pump (начиная с Oracle 10g)

2. Системные таблицы (Data Dictionary)

Словарь данных Oracle — это множество системных таблиц, хранящих метаданные о всех объектах базы данных . Эти таблицы создаются при создании базы данных и обслуживаются исключительно ядром Oracle.

2.1 Основные представления словаря данных

Префикс	Описание
USER_*	Объекты, принадлежащие текущему пользователю
ALL_*	Объекты, доступные текущему пользователю
DBA_*	Все объекты в базе данных (требуются права DBA)
V$*	Динамические представления (Performance Views)
2.2 Важнейшие системные представления

Информация о структуре БД:

sql
-- Таблицы текущего пользователя
SELECT * FROM USER_TABLES;

-- Столбцы таблиц
SELECT * FROM USER_TAB_COLUMNS WHERE TABLE_NAME = 'EMPLOYEES';

-- Индексы
SELECT * FROM USER_INDEXES;

-- Ограничения (constraints)
SELECT * FROM USER_CONSTRAINTS;
Динамические представления (V$) :

V$DATAFILE — файлы данных
V$LOGFILE — файлы redo log
V$CONTROLFILE — файлы управления
V$SESSION — активные сессии
V$SQL — выполняемые SQL-запросы
2.3 Сегменты отката (Undo Segments)

При изменении данных Oracle автоматически помещает предыдущую версию в сегменты отката. Это обеспечивает:

Откат транзакций (ROLLBACK)
Читательскую согласованность (read consistency)
3. Хинты (Hints) в Oracle

3.1 Что такое хинты?

Хинты (подсказки) — это специальные комментарии в SQL-запросах, которые "подсказывают" оптимизатору, как лучше выполнить запрос . Синтаксис:

sql
SELECT /*+ HINT(parameters) */ columns FROM table;
3.2 Зачем нужны хинты?

Иногда оптимизатор Oracle может выбрать неоптимальный план выполнения из-за:

Устаревшей статистики
Неравномерного распределения данных
Сложной структуры запроса 
3.3 Основные типы хинтов

Хинт	                Назначение	                            Пример
INDEX	                Использовать указанный индекс	        /*+ INDEX(employees emp_id_idx) */
NO_INDEX	            Запретить использование индекса	        /*+ NO_INDEX(employees emp_id_idx) */
FULL	                Выполнить полное сканирование таблицы	/*+ FULL(employees) */
LEADING	                Указать порядок соединения таблиц	    /*+ LEADING(e d) */
USE_NL	                Использовать Nested Loops join	        /*+ USE_NL(e d) */
USE_HASH	            Использовать Hash join	                /*+ USE_HASH(e d) */
PARALLEL	            Выполнить запрос в параллель	        /*+ PARALLEL(employees, 4) */
3.4 Пример использования

sql
-- Принудительное использование индекса
SELECT /*+ INDEX(emp emp_id_idx) */ emp_id, emp_name
FROM employees emp
WHERE emp_id = 100;

-- Анализ плана выполнения
EXPLAIN PLAN FOR
SELECT /*+ INDEX(emp emp_id_idx) */ emp_id, emp_name
FROM employees emp
WHERE emp_id = 100;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY());
3.5 Важные замечания

Хинты — это мощный, но опасный инструмент :

Оптимизатор может проигнорировать хинт, если он невозможен
Чрезмерное использование хинтов снижает адаптивность системы
При изменении данных хинт может начать ухудшать производительность

4. Экспорт и импорт данных (Data Pump)

4.1 Общие сведения

Oracle Data Pump (утилиты expdp и impdp) — современный инструмент для логического экспорта/импорта данных, представленный в Oracle 10g .

Преимущества Data Pump:

Значительно быстрее классических exp/imp (до 20 раз)
Поддержка параллельного выполнения (PARALLEL)
Возможность досрочного останова и возобновления
Экспорт/импорт на уровне: база данных → схемы → таблицы → табличные пространства
4.2 Подготовка к работе

Перед использованием Data Pump необходимо создать directory object — объект, указывающий на физическую директорию на сервере :

sql
-- Создание директории
CREATE OR REPLACE DIRECTORY dp_dir AS '/u01/backup';

-- Предоставление прав
GRANT READ, WRITE ON DIRECTORY dp_dir TO hr;
4.3 Команда expdp (экспорт)

bash
# Полный экспорт всей базы данных
expdp system/password DIRECTORY=dp_dir DUMPFILE=full_db.dmp FULL=Y LOGFILE=full_exp.log

# Экспорт схемы пользователя
expdp system/password DIRECTORY=dp_dir DUMPFILE=schema_hr.dmp SCHEMAS=hr LOGFILE=hr_exp.log

# Экспорт конкретных таблиц
expdp hr/hr_password DIRECTORY=dp_dir DUMPFILE=tables.dmp TABLES=employees,departments

# Экспорт только структуры (без данных)
expdp hr/hr_password DIRECTORY=dp_dir DUMPFILE=structure.dmp SCHEMAS=hr CONTENT=METADATA_ONLY
4.4 Команда impdp (импорт)

bash
# Импорт полной базы данных
impdp system/password DIRECTORY=dp_dir DUMPFILE=full_db.dmp FULL=Y

# Импорт схемы с переименованием (remap)
impdp system/password DIRECTORY=dp_dir DUMPFILE=schema_hr.dmp SCHEMAS=hr REMAP_SCHEMA=hr:hr_new

# Импорт с изменением табличного пространства
impdp system/password DIRECTORY=dp_dir DUMPFILE=schema_hr.dmp SCHEMAS=hr REMAP_TABLESPACE=users:new_ts

# Импорт только данных
impdp hr/hr_password DIRECTORY=dp_dir DUMPFILE=tables.dmp TABLES=employees CONTENT=DATA_ONLY
4.5 Полезные параметры

Параметр	Описание
PARALLEL=n	Количество параллельных процессов
EXCLUDE=STATISTICS	Исключить статистику
INCLUDE=TABLE:"IN ('T1','T2')"	Экспортировать только указанные объекты
NETWORK_LINK=link_name	Импорт напрямую из удаленной БД (без промежуточного файла)
TABLE_EXISTS_ACTION=REPLACE	Действие при существовании таблицы
5. Миграция данных с использованием Data Pump

5.1 Типовой сценарий миграции

Шаг 1. Экспорт на исходной БД :

bash
expdp system/password@source_db \
  DIRECTORY=dp_dir \
  DUMPFILE=migration.dmp \
  LOGFILE=exp.log \
  SCHEMAS=hr,oe \
  PARALLEL=4
Шаг 2. Перенос файлов:

bash
scp /u01/backup/migration.dmp user@target_server:/u01/backup/
Шаг 3. Подготовка целевой БД :

sql
-- Создание необходимых tablespace
CREATE TABLESPACE hr_data DATAFILE '/u01/oradata/hr01.dbf' SIZE 1G;
CREATE TABLESPACE oe_data DATAFILE '/u01/oradata/oe01.dbf' SIZE 500M;

-- Создание пользователей
CREATE USER hr IDENTIFIED BY hr_pass DEFAULT TABLESPACE hr_data;
CREATE USER oe IDENTIFIED BY oe_pass DEFAULT TABLESPACE oe_data;

-- Предоставление прав
GRANT CONNECT, RESOURCE TO hr;
GRANT CONNECT, RESOURCE TO oe;
Шаг 4. Импорт на целевой БД :

bash
impdp system/password@target_db \
  DIRECTORY=dp_dir \
  DUMPFILE=migration.dmp \
  LOGFILE=imp.log \
  REMAP_SCHEMA=hr:hr \
  REMAP_SCHEMA=oe:oe \
  PARALLEL=4
5.2 Миграция между разными версиями

При обновлении на более новую версию:

bash
# Экспорт в старой версии
expdp ... VERSION=12.0

# Импорт в новой версии
impdp ... VERSION=LATEST
5.3 Сетевой импорт (без промежуточного файла)

sql
-- На целевой БД создать database link
CREATE DATABASE LINK source_link CONNECT TO system IDENTIFIED BY password USING 'source_db';

-- Импорт напрямую из исходной БД
impdp system/password \
  NETWORK_LINK=source_link \
  SCHEMAS=hr \
  DIRECTORY=dp_dir \
  REMAP_SCHEMA=hr:hr_new
6. Сравнение методов резервного копирования

Характеристика	Физический бэкап (RMAN)	Логический бэкап (expdp)
Уровень	Блоки, файлы	Объекты (таблицы, схемы)
Скорость	Очень высокая	Средняя
Point-in-time recovery	Да	Нет
Выборочное восстановление	Ограниченно	Да
Требования к ARCHIVELOG	Для online backup	Не требуется
Объем бэкапа	Минимальный (только использованные блоки)	Полный размер данных
Кросс-платформенный	Требует конвертации	Да (на уровне данных)
7. Рекомендации и best practices

Всегда тестируйте восстановление — бесполезная резервная копия хуже, чем её отсутствие
Сочетайте физические и логические бэкапы — физические для, логические для выборочного восстановления
Используйте RMAN для физических бэкапов — это рекомендованный инструмент Oracle
Храните бэкапы в разных физических местах (offsite)
Документируйте стратегию — время восстановления (RTO) и точка восстановления (RPO)
Хинты используйте только после анализа execution plan и только когда это действительно необходимо
