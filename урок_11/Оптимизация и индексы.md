Типы индексов в Oracle
1. B-Tree (стандартный индекс)

Определение:
B-Tree (сбалансированное дерево) — это стандартный тип индекса в Oracle, который организует данные в иерархической структуре с корневым блоком, ветвями и листьями. Листовые блоки содержат значения индексированных колонок и ROWID (физический адрес строки). Этот индекс обеспечивает быстрый доступ к данным при поиске по равенству или диапазону.

Синтаксис:

sql
CREATE INDEX index_name ON table_name(column_name);
CREATE INDEX idx_cities_population ON cities(population);
Когда использовать:

Для колонок, которые часто участвуют в операциях сравнения: =, >, <, >=, <=, BETWEEN
Для поиска по префиксу: LIKE 'abc%'
Для колонок с высокой или средней кардинальностью (много уникальных значений)
Для первичных ключей и внешних ключей
Для сортировки (ORDER BY) и группировки (GROUP BY)
2. Unique (уникальный индекс)

Определение:
Уникальный индекс гарантирует, что в индексируемой колонке (или комбинации колонок) не может быть двух строк с одинаковыми значениями. Он автоматически создается при объявлении первичного ключа (PRIMARY KEY) или уникального ограничения (UNIQUE). Помимо обеспечения целостности данных, он также ускоряет поиск по уникальным значениям.

Синтаксис:

sql
CREATE UNIQUE INDEX index_name ON table_name(column_name);
CREATE UNIQUE INDEX idx_country_code ON countries(country_id);
Когда использовать:

Для колонок, которые должны содержать только уникальные значения (например, email, логин, идентификаторы)
Вместо ограничения UNIQUE, когда требуется дополнительный контроль (например, указание табличного пространства)
Для первичных ключей (Oracle автоматически создает уникальный индекс)
Когда важна производительность поиска по уникальному значению
3. Bitmap (битовый индекс)

Определение:
Bitmap индекс хранит битовые карты (bitmaps) для каждого уникального значения индексируемой колонки. Каждый бит в карте соответствует строке таблицы: 1 — значение присутствует, 0 — отсутствует. Такие индексы занимают мало места и эффективны для колонок с небольшим количеством уникальных значений.

Синтаксис:

sql
CREATE BITMAP INDEX index_name ON table_name(column_name);
CREATE BITMAP INDEX idx_cities_is_capital ON cities(is_capital);
Когда использовать:

Для колонок с низкой кардинальностью (менее 1000 уникальных значений)
В хранилищах данных (Data Warehouse) и системах поддержки принятия решений (OLAP)
Когда часто выполняются сложные комбинированные условия с использованием AND, OR, NOT
Не рекомендуется использовать в OLTP-системах с интенсивными операциями INSERT, UPDATE, DELETE (из-за блокировок)
4. Function-Based (индекс на основе функции)

Определение:
Function-Based индекс создается на основе результата вычисления функции или выражения, а не на самой колонке. Это позволяет ускорить запросы, которые используют функции в условиях WHERE или ORDER BY, так как результат функции предварительно вычислен и сохранен в индексе.

Синтаксис:

sql
CREATE INDEX index_name ON table_name(function(column));
CREATE INDEX idx_cities_name_upper ON cities(UPPER(city_name));
Когда использовать:

При поиске с функциями: WHERE UPPER(name) = 'MOSCOW'
При поиске с преобразованиями типов: WHERE TO_CHAR(date_column, 'YYYY') = '2024'
При сортировке по вычисляемым значениям
Для ускорения запросов с LOWER(), TRIM(), арифметическими операциями
Важно: Для использования индекса запрос должен содержать то же самое выражение, что и в определении индекса
5. Composite (составной индекс)

Определение:
Составной (или многоколоночный) индекс включает две и более колонок таблицы. Порядок колонок в определении индекса критически важен: индекс может использоваться только если запрос содержит условие на левый префикс (первую колонку или несколько первых колонок).

Синтаксис:

sql
CREATE INDEX index_name ON table_name(col1, col2, col3);
CREATE INDEX idx_cities_country_pop ON cities(country_id, population);
Когда использовать:

Когда запросы часто фильтруют сразу по нескольким колонкам
Для покрытия условий WHERE, ORDER BY, GROUP BY с несколькими колонками
Когда одна колонка имеет высокую селективность, а другая — низкую
Правило порядка: сначала помещайте колонки с наибольшей селективностью (которые лучше фильтруют данные)
Индекс может использоваться для запросов с условиями на первые 1, 2, 3 и т.д. колонки
6. Reverse Key (индекс с обратным ключом)

Определение:
Reverse Key индекс хранит байты индексируемого значения в обратном порядке. Например, значение 12345 будет храниться как 54321. Это приводит к равномерному распределению записей по всем листовым блокам индекса, устраняя "горячие точки" при вставке последовательных значений.

Синтаксис:

sql
CREATE INDEX index_name ON table_name(column_name) REVERSE;
CREATE INDEX idx_cities_id_rev ON cities(city_id) REVERSE;
Когда использовать:

Для колонок с последовательно возрастающими значениями (например, SERIAL, IDENTITY, последовательности)
Для устранения конкуренции за правый край индекса (right-hand side contention) при высоких нагрузках вставки
В OLTP-системах с интенсивными операциями INSERT
Ограничение: Reverse Key индекс не может использоваться для диапазонных запросов (>, <, BETWEEN), только для точного равенства (=)
7. Descending (индекс с убывающей сортировкой)

Определение:
Descending индекс хранит значения в порядке убывания (по умолчанию B-Tree индексы хранят значения в возрастающем порядке). Это позволяет оптимизировать запросы с сортировкой по убыванию и смешанным сортировкам (одна колонка по возрастанию, другая — по убыванию).

Синтаксис:

sql
CREATE INDEX index_name ON table_name(column_name DESC);
CREATE INDEX idx_cities_pop_desc ON cities(population DESC);
Когда использовать:

Когда запросы часто используют ORDER BY column DESC
Для смешанной сортировки: ORDER BY col1 ASC, col2 DESC
Для ускорения запросов с MAX() или MIN() (особенно в сочетании с другими условиями)
Важно: Можно комбинировать направления для разных колонок в составном индексе
8. Partitioned (секционированный индекс)

Определение:
Partitioned индекс создается на секционированной таблице и может быть двух типов: LOCAL (индекс секционирован по той же схеме, что и таблица) или GLOBAL (индекс секционирован независимо). LOCAL индексы проще в обслуживании и обеспечивают лучшую изоляцию при операциях с секциями.

Синтаксис:

sql
-- LOCAL индекс (секционирован вместе с таблицей)
CREATE INDEX index_name ON table_name(column_name) LOCAL;

-- GLOBAL индекс (секционирован отдельно)
CREATE INDEX index_name ON table_name(column_name) GLOBAL
PARTITION BY RANGE(column_name) (...);
Когда использовать:

Для секционированных таблиц (особенно крупных)
LOCAL индексы подходят для хранилищ данных и для колонок, которые часто используются с ключом секционирования
GLOBAL индексы подходят для уникальных индексов или когда ключ секционирования не входит в условие запроса
Для улучшения управления (partition pruning) — Oracle может исключать неиспользуемые секции из поиска
9. Domain (доменный индекс)

Определение:
Domain индекс — это специализированный тип индекса, который создается для нестандартных типов данных (пространственные, текстовые, XML, мультимедиа) с использованием предопределенных или пользовательских типов индексов (индексных типов). Он реализуется через механизм Oracle Data Cartridge.

Синтаксис:

sql
CREATE INDEX index_name ON table_name(column_name) 
INDEXTYPE IS index_type_name [PARAMETERS 'parameters'];

-- Пример для пространственных данных (Oracle Spatial)
CREATE INDEX idx_spatial ON locations(location) 
INDEXTYPE IS MDSYS.SPATIAL_INDEX;

-- Пример для текстового поиска (Oracle Text)
CREATE INDEX idx_text ON documents(content) 
INDEXTYPE IS CTXSYS.CONTEXT;
Когда использовать:

Для полнотекстового поиска (Oracle Text)
Для пространственных/географических запросов (Oracle Spatial)
Для XML-данных (XMLIndex)
Для поиска в JSON-документах
Для мультимедийных данных (изображения, аудио, видео через Oracle Multimedia)
10. Invisible (невидимый индекс)

Определение:
Invisible индекс — это индекс, который существует в базе данных, поддерживается при операциях DML (вставка, обновление, удаление), но не учитывается оптимизатором запросов при построении плана выполнения. Он позволяет тестировать влияние индекса на производительность без его фактического удаления или риска нарушения работы существующих приложений.

Синтаксис:

sql
-- Создание невидимого индекса
CREATE INDEX index_name ON table_name(column_name) INVISIBLE;

-- Изменение видимости существующего индекса
ALTER INDEX index_name VISIBLE;
ALTER INDEX index_name INVISIBLE;
Когда использовать:

Для тестирования нового индекса перед его включением в production
Для проверки влияния индекса на производительность вставки/обновления
Для временного отключения индекса (например, при проблемах с производительностью)
Для поэтапного внедрения изменений в индексацию
Важно: Невидимые индексы все еще занимают место и обновляются при DML-операциях
Это описание покрывает основные типы индексов в Oracle Database. Каждый тип имеет свою нишу применения, и выбор правильного типа индекса — ключевой фактор в оптимизации производительности запросов.


Oracle и составные индексы:

Индекс может использоваться только если условие содержит левый префикс
INDEX (country_id, importance, population) будет работать для:

WHERE country_id = 'US'
WHERE country_id = 'US' AND importance > 5
WHERE country_id = 'US' AND importance > 5 AND population < 1000000
НЕ будет работать для:

WHERE importance > 5 (пропущен country_id)
WHERE population < 1000000 (пропущены первые две колонки)
Skip Scan (Oracle 9i+):

Позволяет использовать составной индекс даже без префикса (если префикс имеет мало уникальных значений)
