1. NESTED LOOPS (ВЛОЖЕННЫЕ ЦИКЛЫ)

Как работает

text
Алгоритм:

FOR каждой строки row1 в таблице A:
    FOR каждой строки row2 в таблице B:
        IF row1.key = row2.key:
            вернуть (row1, row2)
        END IF
    END FOR
END FOR
Визуализация

text
Таблица A (100 строк)          Таблица B (1000 строк)
┌────┬─────────┐               ┌────┬─────────┐
│ id │ name    │               │ id │ city    │
├────┼─────────┤               ├────┼─────────┤
│ 1  │ Moscow  │──────────────→│ 1  │ Moscow  │
│ 2  │ London  │──┐            │ 1  │ Moscow  │
│ 3  │ Paris   │  │            │ 2  │ London  │
└────┴─────────┘  │            │ 3  │ Paris   │
                  │            │ 3  │ Paris   │
                  │            └────┴─────────┘
                  │
                  └─ 100 × 1000 = 100,000 итераций (без индекса)
                     с индексом: 100 × log₂(1000) ≈ 1000 итераций
Сложность (Big O)




Сценарий	        Сложность	        Пояснение
Без индекса	        O(N × M)	        Полное сканирование правой таблицы для каждой строки левой
С индексом	        O(N × log M)	    Поиск по B-Tree индексу за логарифмическое время
Лучший случай	    O(N × 1)	        Уникальный индекс, найдено сразу
Худший случай	    O(N × M)	        Нет индекса, большие таблицы
Когда используется NESTED LOOPS

Оптимальные сценарии:

Одна таблица маленькая, другая большая с индексом

sql
-- Маленькая таблица (10 записей) JOIN с большой (1M записей)
-- Индекс на внешнем ключе в большой таблице

SELECT * FROM departments d    -- 10 строк
JOIN employees e ON d.dept_id = e.dept_id;  -- индекс на e.dept_id

Нужно вернуть небольшое количество строк (< 10,000)
Соединение по уникальному индексу
Есть возможность "ранней остановки" (FIRST_ROWS оптимизация)

Плохие сценарии:
Обе таблицы большие (100K × 100K = 10 млрд операций)
Нет подходящего индекса на правой таблице
Возвращается много строк (> 20% от таблицы)

Пример в Oracle (принудительное использование)

sql
-- Подсказка для использования NESTED LOOPS
SELECT /*+ USE_NL(t1 t2) LEADING(t1) */ *
FROM cities t1
JOIN weather_data t2 ON t1.city_name = t2.city_name
WHERE t1.population > 5000000;

2. HASH JOIN

Как работает

text
Алгоритм:

1. BUILD PHASE (Построение хеш-таблицы):
   - Выбираем меньшую таблицу как "build table"
   - Для каждой строки вычисляем хеш от ключа соединения
   - Сохраняем в хеш-таблице (в памяти или на диске)

2. PROBE PHASE (Проверка):
   - Проходим по большей таблице ("probe table")
   - Для каждой строки вычисляем хеш
   - Ищем совпадение в хеш-таблице
   - Если нашли - возвращаем строку
Визуализация

text
┌─────────────────────────────────────────────────────────────────────┐
│  BUILD PHASE                    PROBE PHASE                         │
│                                                                      │
│  Таблица A (меньшая)            Таблица B (большая)                 │
│  ┌────┬─────────┐               ┌────┬─────────┐                   │
│  │ id │ name    │               │ id │ city    │                   │
│  ├────┼─────────┤               ├────┼─────────┤                   │
│  │ 1  │ Moscow  │──→ hash(1)    │ 2  │ London  │──→ hash(2) ──┐    │
│  │ 2  │ London  │    ↓          │ 1  │ Moscow  │──→ hash(1) ──┤    │
│  │ 3  │ Paris   │    ↓          │ 3  │ Paris   │──→ hash(3) ──┤    │
│  └────┴─────────┘    ↓          │ 1  │ Moscow  │──→ hash(1) ──┤    │
│                      ↓          └────┴─────────┘                ↓  │
│               ┌─────────────┐                                   ↓  │
│               │ ХЕШ-ТАБЛИЦА │                              ┌────┴────┐
│               │ hash(1) → 1 │←─────────────────────────────┘         │
│               │ hash(2) → 2 │←──────────────────────────────────┐    │
│               │ hash(3) → 3 │←─────────────────────────────────┐│    │
│               └─────────────┘                                  ││    │
│                                                           Результат │
└─────────────────────────────────────────────────────────────────────┘
Сложность

Сценарий	                Сложность	        Пояснение
В памяти	                O(N + M)	        Оба прохода линейные
С переполнением (на диск)	O((N+M) × K)	    K - количество проходов
Лучший случай	            O(N + M)	        Вся хеш-таблица в памяти
Худший случай	            O(N × M)	        Много коллизий, переполнение
Когда используется HASH JOIN

Оптимальные сценарии:
Обе таблицы большие (100K+ строк каждая)
Нет подходящего индекса или индекс неэффективен
Нужно вернуть много строк (> 10-20% от таблицы)
Соединение не по уникальному ключу (много совпадений)
Нет требования к порядку сортировки результата

Плохие сценарии:
Одна таблица очень маленькая (тогда Nested Loops лучше)
Нехватка памяти (хеш-таблица не помещается в RAM, будут чтения с диска)
Соединение по неравенству (<, >, between) — HASH JOIN работает только на равенство
Пример в Oracle

sql
-- Принудительное использование HASH JOIN
SELECT /*+ USE_HASH(c w) */ *
FROM cities c
JOIN weather_data w ON c.city_name = w.city_name
WHERE w.temp_c > 25;

-- Oracle сам выбирает HASH JOIN для больших таблиц
SELECT c.city_name, COUNT(*)
FROM cities c
JOIN weather_data w ON c.city_name = w.city_name
GROUP BY c.city_name;

3. SORT MERGE JOIN

Как работает

text
Алгоритм:

1. SORT PHASE (Сортировка):
   - Сортируем обе таблицы по ключу соединения
   - Если данные уже отсортированы (по индексу) - пропускаем

2. MERGE PHASE (Слияние):
   - Проходим по обоим отсортированным наборам
   - Как при слиянии двух отсортированных массивов
   - Сдвигаем указатель в той таблице, где ключ меньше
Визуализация

text
ШАГ 1: СОРТИРОВКА
─────────────────────────────────────────────────────────────────
Таблица A (после сортировки)    Таблица B (после сортировки)
┌────┬─────────┐                ┌────┬─────────┐
│ id │ name    │                │ id │ city    │
├────┼─────────┤                ├────┼─────────┤
│ 1  │ Moscow  │                │ 1  │ Moscow  │
│ 2  │ London  │                │ 1  │ Moscow  │
│ 3  │ Paris   │                │ 2  │ London  │
│ 4  │ Berlin  │                │ 3  │ Paris   │
│ 5  │ Madrid  │                │ 3  │ Paris   │
└────┴─────────┘                │ 4  │ Berlin  │
                                │ 5  │ Madrid  │
                                └────┴─────────┘

ШАГ 2: MERGE (СЛИЯНИЕ)
─────────────────────────────────────────────────────────────────
Указатель A → id=1    Указатель B → id=1    → СОВПАДЕНИЕ!
Указатель A → id=1    Указатель B → id=1    → СОВПАДЕНИЕ!
Указатель A → id=1    Указатель B → id=2    → id=1 < 2, двигаем A
Указатель A → id=2    Указатель B → id=2    → СОВПАДЕНИЕ!
Указатель A → id=2    Указатель B → id=3    → id=2 < 3, двигаем A
...
Сложность

Сценарий	        Сложность	            Пояснение
Сортировка	        O(N log N + M log M)	Если данные не отсортированы
Слияние	            O(N + M)	            Один проход по обоим наборам
Общая (худший)	    O(N log N + M log M)	Доминирует сортировка
Общая (лучший)	    O(N + M)	            Если данные уже отсортированы
Когда используется SORT MERGE JOIN

Оптимальные сценарии:

Соединение по неравенству (<, >, <=, >=, BETWEEN)

sql
-- SORT MERGE JOIN отлично подходит для диапазонов
SELECT * FROM employees e
JOIN salaries s ON e.emp_id = s.emp_id
WHERE s.salary BETWEEN 30000 AND 50000;

Обе таблицы уже отсортированы (по индексу)
Нет возможности использовать HASH JOIN (память ограничена)
Требуется отсортированный результат (экономит ORDER BY)
Соединение больших таблиц без индексов

Плохие сценарии:
Нужно вернуть небольшое количество строк (Nested Loops эффективнее)
Данные не отсортированы и сортировка дорогая
Много дубликатов в ключах соединения
Пример в Oracle

sql
-- Принудительное использование SORT MERGE JOIN
SELECT /*+ USE_MERGE(c w) */ *
FROM cities c
JOIN weather_data w ON c.city_name = w.city_name
WHERE c.population > 1000000;

-- Для неравенства Oracle часто выбирает MERGE JOIN
SELECT * FROM cities c
JOIN regions r ON c.country_id = r.country_id
WHERE c.population > r.population;  -- неравенство!

Требования к памяти	           Минимальные	                Высокие (хеш-таблица)	Средние (сортировка)

Поддержка неравенств	        Нет	                        Нет	                     Да

Худший сценарий	            Большая × Большая (без индекса)	Нехватка памяти	        Неотсортированные данные

Ранняя остановка	        Да	                            Нет	                    Нет

КАК ВЫБИРАЕТ ОПТИМИЗАТОР?

Oracle CBO (Cost-Based Optimizer) оценивает:

text
COST(JOIN) = 
    - Размер таблиц (статистика)
    - Наличие индексов
    - Селективность условий (сколько строк вернется)
    - Доступная память (PGA)
    - Требования к сортировке
Примеры выбора оптимизатором

sql
-- 1. NESTED LOOPS: маленькая таблица + индекс
SELECT * FROM countries cnt          -- 40 строк
JOIN cities c ON cnt.country_id = c.country_id  -- индекс на c.country_id
WHERE cnt.continent = 'Europe';

-- 2. HASH JOIN: две большие таблицы, равенство
SELECT * FROM cities c               -- 100K строк
JOIN weather_data w ON c.city_name = w.city_name;  -- 1M строк

-- 3. SORT MERGE JOIN: неравенство
SELECT * FROM cities c
JOIN regions r ON c.country_id = r.country_id
WHERE c.population > r.population;   -- неравенство!

-- 4. HASH JOIN RIGHT: для LEFT JOIN с большими таблицами
SELECT * FROM countries cnt
LEFT JOIN cities c ON cnt.country_id = c.country_id;

Вопрос	Ответ
Что такое Nested Loops?	Алгоритм с двойным циклом: для каждой строки из A ищем в B (обычно по индексу). Эффективен когда A маленькая, B большая с индексом.
Что такое Hash Join?	Строит хеш-таблицу по меньшей таблице, затем проходится по большей. Лучший выбор для двух больших таблиц.
Что такое Sort Merge Join?	Сортирует обе таблицы по ключу, затем сливает. Единственный алгоритм, поддерживающий неравенства (<, >, between).
Какой алгоритм лучше?	Зависит от данных. Нет "лучшего", каждый хорош в своих сценариях. Оптимизатор выбирает по стоимости.
Как увидеть план?	EXPLAIN PLAN FOR ... SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
