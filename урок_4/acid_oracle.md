Введение в ACID и транзакции Oracle
В Oracle транзакция определяется как логическая, атомарная единица работы, состоящая из одного или нескольких SQL-операторов . Все транзакции в Oracle строго соответствуют ACID-свойствам, что является одним из ключевых отличий СУБД от файловых систем . Каждой транзакции присваивается уникальный идентификатор (transaction ID) .

Реализация свойств ACID в Oracle
Атомарность (Atomicity)
Реализация в Oracle: Обеспечивается через журнал отката (Undo Log) и механизм фиксации (COMMIT) / отката (ROLLBACK). Oracle гарантирует атомарность на двух уровнях: на уровне всей транзакции и на уровне отдельного оператора.

Принцип работы:

Когда транзакция начинает изменять данные, Oracle прежде всего записывает в Undo-сегменты информацию, необходимую для отмены этих изменений (старые данные) .
Если транзакция завершается командой COMMIT, все изменения фиксируются как постоянные. Если выполняется ROLLBACK (полный или до сохраненной точки SAVEPOINT), Oracle считывает информацию из Undo-сегментов и восстанавливает данные в исходное состояние .
Атомарность на уровне оператора: Даже внутри транзакции отдельный SQL-оператор выполняется атомарно. Если в процессе выполнения многорядной операции (например, UPDATE 100 rows) происходит ошибка, Oracle автоматически откатывает изменения только этого оператора, но предыдущие успешные операторы в транзакции остаются незафиксированными .
Итог: Либо все изменения транзакции применяются, либо ни одно из них.

Согласованность (Consistency)
Реализация в Oracle: Достигается сочетанием механизмов: многоверсионности (MVCC), системного номера изменения (SCN) и ограничений целостности (constraints).

Принцип работы:

Транзакция всегда переводит базу данных из одного согласованного состояния в другое. Это означает, что все ограничения (PRIMARY KEY, CHECK, FOREIGN KEY и др.), триггеры и правила данных соблюдаются к моменту фиксации .
Ключевую роль играет System Change Number (SCN) — логический внутренний таймер, который монотонно увеличивается и упорядочивает все события в базе данных . Каждая транзакция и каждый коммит получают свой уникальный SCN.
Если транзакция пытается нарушить ограничение (например, вставить NULL в NOT NULL поле или отрицательное значение в столбец с CHECK constraint), она прерывается ошибкой .
Итог: Целостность данных гарантируется на всем протяжении транзакции.

Изоляция (Isolation)
Это, пожалуй, самая технически сложная часть, реализованная через многоверсионность (MVCC) и блокировки.

Реализация в Oracle: Фундамент изоляции — MVCC (Multiversion Concurrency Control) .

Принцип работы:

Когда транзакция изменяет блок данных, оригинальные данные (на момент "до изменения") сохраняются в Undo-сегменты. Сам же блок данных в оперативной памяти (и на диске) содержит уже измененные данные .
Другая транзакция, читающая эти же данные, не видит незафиксированных изменений. Вместо этого Oracle, используя SCN и Undo-данные, строит для читающей транзакции "согласованный по времени срез" (consistent read snapshot) данных на нужный момент .
Благодаря этому читающие транзакции никогда не блокируют пишущие, а пишущие не блокируют читающие. Блокировки накладываются только в случае попытки двумя транзакциями изменить одни и те же данные .
Уровни изоляции: Oracle поддерживает два основных уровня, соответствующих стандарту SQL:
READ COMMITTED (по умолчанию): Каждый запрос в транзакции видит только те данные, которые были зафиксированы до момента начала этого запроса .
SERIALIZABLE: Транзакция видит данные только на момент своего начала. Изменения, сделанные другими транзакциями после ее старта, для нее невидны .
Также есть режим READ ONLY, который не входит в стандарт, но гарантирует видимость данных только на момент начала транзакции и запрещает любые изменения .
Итог: Обеспечивается строгая изоляция и консистентность чтения без ущерба для параллелизма.

Долговечность (Durability)
Реализация в Oracle: Обеспечивается журналом повторного выполнения (Redo Log) и механизмами восстановления (Instance Recovery).

Принцип работы:

Когда транзакция выполняет COMMIT, Oracle не сбрасывает немедленно измененные блоки данных на диск. Это было бы слишком медленно.
Вместо этого Oracle записывает в буфер Redo Log всю информацию об изменениях, сделанных транзакцией. В момент COMMIT процесс LGWR (Log Writer) немедленно сбрасывает этот буфер на диск в файлы Redo Log.
Только после того, как запись в Redo Log физически оказалась на диске, управление возвращается пользователю с сообщением об успешном COMMIT .
Если после этого произойдет сбой питания, при перезапуске базы данных процесс восстановления (SMON) прочитает Redo Log и повторно применит (или откатит) все изменения, гарантируя, что ни одна закоммиченная транзакция не будет потеряна.
Итог: Как только пользователь получил подтверждение о COMMIT, его данные гарантированно сохранены и переживут любой сбой.

Внутренние механизмы Oracle для ACID
Системный номер изменения (SCN)
SCN — это сердце ACID в Oracle. Это внутренние "часы" базы данных. SCN присваивается каждому событию (начало транзакции, изменение строки, коммит). Он используется для:

Определения точки согласованности при чтении (какие версии данных видит транзакция).

Упорядочивания событий восстановления.

Синхронизации в RAC (Real Application Clusters) .

Журнал отката (Undo)
Хранит "старые" версии данных.

Роль в ACID:

Атомарность: Для отката транзакций (ROLLBACK).

Изоляция: Для построения "согласованных срезов" (consistent read) для читающих транзакций.

Согласованность: Обеспечивает чтение без блокировок.
Когда изменяется строка в блоке, Oracle записывает в Undo сегмент информацию, как вернуть эту строку в исходное состояние .

Журнал повторного выполнения (Redo)
Хранит "историю" всех изменений. Это журнал, который пишется последовательно. Его главная задача — Долговечность.

Роль в ACID: При сбое Redo Log используется для "проигрывания" всех закоммиченных транзакций заново, чтобы восстановить данные, которые не успели записаться из буферного кэша на диск.

Блокировки (Locks)
Хотя MVCC решает проблему чтения, для изоляции записи нужны блокировки.

Oracle использует блокировки на уровне строк (row-level locks). Когда транзакция изменяет строку, она блокирует ее для других изменяющих транзакций до своего завершения (COMMIT/ROLLBACK). Это гарантирует, что две транзакции не обновят одну строку одновременно .
