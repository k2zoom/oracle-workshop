1. ОПРЕДЕЛЕНИЕ
Временная таблица (Temporary Table) — это специальный тип таблиц, предназначенный для хранения промежуточных данных в течение сессии или транзакции. Данные во временных таблицах видны только текущей сессии и автоматически удаляются по окончании сессии или транзакции.

2. НАЗНАЧЕНИЕ ВРЕМЕННЫХ ТАБЛИЦ
Хранение промежуточных результатов сложных запросов
Улучшение производительности при многоэтапной обработке данных
Разбиение сложных операций на несколько простых шагов
Хранение данных, не требующих постоянного хранения
Организация рабочих областей для приложений
Избежание повторных дорогостоящих вычислений
Решение проблем с мутирующими таблицами в триггерах

3. ОСОБЕННОСТИ ВРЕМЕННЫХ ТАБЛИЦ В ORACLE
3.1 Принципиальные отличия от постоянных таблиц
Характеристика	        Постоянная таблица	        Временная таблица
Хранение данных	        В табличном пространстве	В временном табличном пространстве (TEMP)
Видимость	            Все сессии	                Только своя сессия
Длительность	        Постоянно	                До конца сессии/транзакции
Конкуренция	            Требуется синхронизация	    Нет конфликтов между сессиями
REDO логи	            Генерируется	            Минимум или отсутствует
UNDO логи	            Генерируется	            Минимум
Бэкап/восстановление	Требуется	                Не требуется

3.2 Важные особенности Oracle
Структура таблицы постоянна — создается один раз, данные временные
Данные приватны — каждая сессия видит только свои данные
Определение глобально — все сессии могут использовать структуру
Хранилище в TEMP tablespace — не занимает место в permanent tablespace
Минимальная генерация REDO — только для отмены (UNDO)
Нет блокировок между сессиями — каждая работает со своим экземпляром данных

4. ТИПЫ ВРЕМЕННЫХ ТАБЛИЦ
4.1 Transaction-Specific (ON COMMIT DELETE ROWS)
Данные существуют только в течение текущей транзакции. После COMMIT или ROLLBACK данные автоматически удаляются.

Характеристики:

Данные видны только в рамках одной транзакции
После COMMIT данные очищаются
Идеально для промежуточных вычислений внутри транзакции
Наиболее часто используемый тип

4.2 Session-Specific (ON COMMIT PRESERVE ROWS)
Данные сохраняются на всю продолжительность сессии. При COMMIT данные не удаляются, они доступны до завершения сессии (logout) или явного удаления (TRUNCATE/DELETE).

Характеристики:
Данные живут всю сессию
COMMIT не очищает данные
Можно использовать в нескольких транзакциях одной сессии
Полезно для кэширования данных на время работы пользователя

5. СИНТАКСИС СОЗДАНИЯ
5.1 Транзакционная временная таблица
sql
CREATE GLOBAL TEMPORARY TABLE имя_таблицы (
    колонка1 тип_данных [ограничения],
    колонка2 тип_данных [ограничения],
    ...
) ON COMMIT DELETE ROWS;

5.2 Сессионная временная таблица
sql
CREATE GLOBAL TEMPORARY TABLE имя_таблицы (
    колонка1 тип_данных [ограничения],
    колонка2 тип_данных [ограничения],
    ...
) ON COMMIT PRESERVE ROWS;

5.3 С индексами и ограничениями
sql
CREATE GLOBAL TEMPORARY TABLE temp_orders (
    id NUMBER,
    order_date DATE,
    customer_id NUMBER,
    amount NUMBER(10,2),
    CONSTRAINT pk_temp_orders PRIMARY KEY (id)
) ON COMMIT DELETE ROWS;

CREATE INDEX idx_temp_orders_date ON temp_orders(order_date);

6. ОПЕРАЦИИ С ВРЕМЕННЫМИ ТАБЛИЦАМИ
6.1 INSERT
sql
INSERT INTO temp_orders (id, order_date, customer_id, amount)
SELECT id, order_date, customer_id, total_amount
FROM orders
WHERE order_date > SYSDATE - 7;
6.2 SELECT
sql
SELECT * FROM temp_orders WHERE amount > 1000;
6.3 UPDATE
sql
UPDATE temp_orders SET amount = amount * 1.1 WHERE customer_id = 123;
6.4 DELETE
sql
DELETE FROM temp_orders WHERE amount < 500;
6.5 TRUNCATE
sql
TRUNCATE TABLE temp_orders;  -- явная очистка данных
6.6 DROP
sql
DROP TABLE temp_orders;  -- удаление структуры

7. ПОВЕДЕНИЕ ПРИ РАЗНЫХ ОПЕРАЦИЯХ
7.1 ON COMMIT DELETE ROWS
Операция	Результат
INSERT, затем COMMIT	Данные удаляются после COMMIT
INSERT, затем ROLLBACK	Данные удаляются после ROLLBACK
INSERT, затем SELECT	Данные видны до COMMIT/ROLLBACK
TRUNCATE	Немедленная очистка данных
Конец сессии	Данные автоматически очищаются
7.2 ON COMMIT PRESERVE ROWS
Операция	Результат
INSERT, затем COMMIT	Данные СОХРАНЯЮТСЯ
INSERT, затем ROLLBACK	Данные откатываются (как в обычной таблице)
INSERT, затем SELECT	Данные видны
Несколько транзакций	Данные накапливаются между транзакциями
Конец сессии	Данные автоматически очищаются

8. ИНДЕКСЫ И ОГРАНИЧЕНИЯ
8.1 Поддерживаемые объекты
Во временных таблицах можно создавать:

Индексы (B-tree, bitmap, function-based)

PRIMARY KEY, FOREIGN KEY

UNIQUE, CHECK constraints

Триггеры

8.2 Особенности
Индексы также временные (существуют, пока существует таблица)
Ограничения проверяются только для данных текущей сессии
FOREIGN KEY могут ссылаться на постоянные таблицы
Нельзя ссылаться из постоянных таблиц на временные (внешние ключи)

9. ПРОИЗВОДИТЕЛЬНОСТЬ
9.1 Преимущества
Нет генерации REDO для данных (только для отмены)
Минимум UNDO — только для отката изменений
Нет журналирования — быстрее вставка/обновление
Нет конкуренции — каждая сессия работает со своим набором данных
Операции в памяти — данные могут полностью помещаться в памяти

9.2 Когда использовать
Промежуточные наборы данных (ETL процессы)
Многоэтапные расчеты с большими объемами
Кэширование результатов сложных запросов в сессии
Обработка данных, не требующих постоянного хранения
Временные хранилища для отчетов

11. ПРИМЕНЕНИЕ В РЕАЛЬНЫХ ЗАДАЧАХ
11.1 В ETL процессах
sql
-- Этап 1: Загрузить сырые данные во временную таблицу
INSERT INTO temp_raw_data SELECT ... FROM внешний_источник;

-- Этап 2: Трансформация
UPDATE temp_raw_data SET ...;

-- Этап 3: Загрузка в целевую таблицу
INSERT INTO target_table SELECT ... FROM temp_raw_data;
COMMIT;  -- для ON COMMIT DELETE ROWS данные очистятся автоматически
11.2 В триггерах (обход мутирующих таблиц)
sql
CREATE OR REPLACE TRIGGER trg_avoid_mutating
FOR INSERT ON orders
COMPOUND TRIGGER
    -- Временная таблица уже создана глобально
    
    BEFORE STATEMENT IS
    BEGIN
        -- Очищаем временную таблицу для новой операции
        DELETE FROM temp_order_log;
    END BEFORE STATEMENT;
    
    AFTER EACH ROW IS
    BEGIN
        -- Сохраняем ID новых заказов во временную таблицу
        INSERT INTO temp_order_log (order_id) VALUES (:NEW.id);
    END AFTER EACH ROW;
    
    AFTER STATEMENT IS
    BEGIN
        -- Теперь можно читать orders, так как мы не в строчном триггере
        UPDATE orders_summary SET total_orders = (
            SELECT COUNT(*) FROM temp_order_log
        );
    END AFTER STATEMENT;
END;
/

11.3 Для сложных отчетов
sql
-- Шаг 1: Агрегация данных во временной таблице
INSERT INTO temp_sales_summary
SELECT product_id, SUM(amount) as total_sales
FROM orders
GROUP BY product_id;

-- Шаг 2: Добавление дополнительных расчетов
UPDATE temp_sales_summary t
SET rank = (
    SELECT COUNT(*) + 1
    FROM temp_sales_summary t2
    WHERE t2.total_sales > t.total_sales
);

-- Шаг 3: Финальный отчет
SELECT * FROM temp_sales_summary ORDER BY rank;

13. ОГРАНИЧЕНИЯ И ОСОБЕННОСТИ
13.1 Ограничения
Нельзя использовать в внешних ключах, ссылающихся на постоянные таблицы (но можно из постоянных на временные)
Статистика не собирается автоматически (может влиять на оптимизатор)
Не поддерживаются материализованные представления
Не участвуют в бэкапах и восстановлении
Не могут быть партиционированными
Не поддерживают параллельное выполнение в некоторых версиях

13.2 Особенности
TRUNCATE на временной таблице очищает данные только текущей сессии
DROP TABLE удаляет структуру для всех сессий
Индексы автоматически удаляются при DROP TABLE
Можно создавать несколько временных таблиц с одинаковыми именами в разных схемах
Поддерживаются все типы данных, включая LOB

14. СРАВНЕНИЕ С ДРУГИМИ МЕХАНИЗМАМИ
Механизм	Хранение	Видимость	Длительность	Использование

15. ТЕРМИНОЛОГИЯ
Термин	Определение
GLOBAL TEMPORARY TABLE	Временная таблица с глобальным определением
ON COMMIT DELETE ROWS	Данные удаляются после COMMIT/ROLLBACK
ON COMMIT PRESERVE ROWS	Данные сохраняются между транзакциями
TEMP tablespace	Специальное табличное пространство для временных данных
Сессия (Session)	Подключение пользователя к БД
Транзакция (Transaction)	Логическая единица работы с COMMIT/ROLLBACK

16. ПРАКТИЧЕСКИЕ РЕКОМЕНДАЦИИ
16.1 Когда использовать
Многоэтапная обработка больших объемов данных
Необходимость временного хранения результатов сложных запросов
Разбиение сложных операций на управляемые части
Кэширование данных в рамках сессии пользователя
Обход ограничений мутирующих таблиц в триггерах

16.2 Когда НЕ использовать
Данные нужны дольше, чем одна сессия
Данные должны быть доступны другим пользователям
Небольшие объемы данных (CTE может быть достаточно)
Простые однократные запросы

16.3 Best Practices
Всегда указывать ON COMMIT в соответствии с требованиями
Создавать индексы на часто используемые колонки
Использовать TRUNCATE вместо DELETE для быстрой очистки
Следить за размером временного табличного пространства
Не создавать временные таблицы в каждой сессии без необходимости
Использовать осмысленные имена (префикс TEMP_)
