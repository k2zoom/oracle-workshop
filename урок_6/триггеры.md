Триггер (Trigger) — это хранимая процедура особого типа, которая автоматически выполняется (срабатывает) при наступлении определенного события в базе данных.


Триггеры не вызываются явно пользователем или приложением — они запускаются автоматически при выполнении операций DML (INSERT, UPDATE, DELETE) или DDL (CREATE, ALTER, DROP) над указанными объектами базы данных.

НАЗНАЧЕНИЕ ТРИГГЕРОВ:

Обеспечение сложных ограничений целостности данных
Автоматическое ведение журналов изменений (аудит)
Поддержание согласованности данных (каскадные обновления)
Автоматическая генерация значений (например, первичных ключей)
Запрет некорректных операций
Репликация данных
Вызов внешних приложений при изменениях в БД

Классификация триггеров

По типу события
Тип	                Описание
DML-триггеры	    Срабатывают при INSERT, UPDATE, DELETE
DDL-триггеры	    Срабатывают при CREATE, ALTER, DROP и др.
Системные триггеры	Срабатывают при событиях БД (запуск, остановка, логин, логаут)

По времени срабатывания
Время	        Описание
BEFORE	        До выполнения операции DML
AFTER	        После выполнения операции DML
INSTEAD OF	    Вместо операции DML (для представлений)

По уровню срабатывания
Уровень	            Описание
FOR EACH ROW	    Срабатывает для каждой изменяемой строки
FOR EACH STATEMENT	Срабатывает один раз на операцию DML

По составу событий
Тип	                    Описание
Простые	                На одно событие (INSERT, UPDATE или DELETE)
Составные	            На несколько событий (INSERT OR UPDATE и т.д.)
С составным триггером	Специальная конструкция с несколькими точками срабатывания

Синтаксис создания триггера

CREATE [OR REPLACE] TRIGGER имя_триггера
    [BEFORE | AFTER | INSTEAD OF]
    [INSERT | UPDATE [OF колонка] | DELETE | ...]
    [OR INSERT | OR UPDATE | OR DELETE ...]
    ON [схема.]таблица_или_представление
    [REFERENCING OLD AS old NEW AS new]
    [FOR EACH ROW]
    [WHEN (условие)]
    [FOLLOWS другой_триггер]
    [ENABLE | DISABLE]
DECLARE
    -- объявление переменных (опционально)
BEGIN
    -- тело триггера (PL/SQL код)
END;


ПСЕВДОЗАПИСИ :OLD И :NEW
Определение
:OLD и :NEW — специальные псевдозаписи, предоставляющие доступ к значениям строк до и после выполнения DML-операции в строчных триггерах (FOR EACH ROW).

Содержимое по типу операции
Операция	        :OLD	                :NEW
INSERT	            NULL (все поля)	        Вставляемые значения
UPDATE	            Значения до обновления	Значения после обновления
DELETE	            Значения до удаления	NULL (все поля)
Используются только в строчных триггерах (FOR EACH ROW)

В BEFORE-триггерах можно изменять значения :NEW (кроме для DELETE)
В AFTER-триггерах :NEW изменять нельзя
:OLD изменять нельзя никогда


В теле триггера можно определить, какое событие вызвало срабатывание:

Функция	                Возвращает
INSERTING	            TRUE, если операция INSERT
UPDATING	            TRUE, если операция UPDATE
UPDATING('колонка')	    TRUE, если обновляется указанная колонка
DELETING	            TRUE, если операция DELETE


ПОРЯДОК ВЫПОЛНЕНИЯ ТРИГГЕРОВ
Для одной строки
BEFORE STATEMENT триггеры
BEFORE EACH ROW триггеры
Выполнение самой DML-операции
AFTER EACH ROW триггеры
AFTER STATEMENT триггеры

Для нескольких строк
Цикл шагов 2-4 для каждой изменяемой строки.

INSTEAD OF триггеры
Выполняются вместо самой DML-операции, полностью заменяя ее.

МУТИРУЮЩИЕ ТАБЛИЦЫ
Определение
Мутирующая таблица — таблица, которая находится в состоянии изменения (выполняется DML-операция), и к которой триггер пытается обратиться.

Ограничение
Строчные триггеры (FOR EACH ROW) не могут читать или изменять мутирующую таблицу, так как это может привести к неопределенному состоянию данных или бесконечным циклам.

Исключение
Запросы к мутирующей таблице возможны только в триггерах уровня STATEMENT или с использованием специальных методов (компаунд-триггеры, временные таблицы).

ВИДЫ ТРИГГЕРОВ
DML-триггеры
Самый распространенный тип. Срабатывают при операциях INSERT, UPDATE, DELETE над таблицами или представлениями.


INSTEAD OF триггеры
Специальный тип триггеров для представлений (VIEW). Позволяют выполнять DML-операции над представлениями, которые обычно не поддерживают изменения (например, с JOIN).


DDL-триггеры
Срабатывают при выполнении DDL-команд (CREATE, ALTER, DROP, TRUNCATE и др.). Могут быть на уровне схемы или базы данных.


Системные триггеры
Срабатывают при событиях экземпляра базы данных:
STARTUP — запуск БД
SHUTDOWN — остановка БД
LOGON — вход пользователя
LOGOFF — выход пользователя
SERVERERROR — ошибка сервера


Компаунд-триггеры (составные)
Специальная структура, объединяющая несколько точек срабатывания в одном триггере:
BEFORE STATEMENT
BEFORE EACH ROW
AFTER EACH ROW
AFTER STATEMENT
Позволяют обойти ограничения мутирующих таблиц и улучшить производительность.


АТРИБУТЫ ТРИГГЕРОВ
FOLLOWS
Определяет порядок выполнения триггеров одного типа. Триггер с FOLLOWS выполняется после указанного триггера.

ENABLE / DISABLE
Управление состоянием триггера:
ENABLE — триггер активен (по умолчанию)
DISABLE — триггер временно отключен

WHEN (условие)
Позволяет задать дополнительное условие для выполнения триггера. Условие проверяется для каждой строки (только для строчных триггеров).

REFERENCING
Позволяет переименовать псевдозаписи :OLD и :NEW:
sql
REFERENCING OLD AS старая_запись NEW AS новая_запись

УПРАВЛЕНИЕ ТРИГГЕРАМИ
Включение/отключение
sql
ALTER TRIGGER имя_триггера ENABLE;
ALTER TRIGGER имя_триггера DISABLE;
ALTER TABLE имя_таблицы ENABLE ALL TRIGGERS;
ALTER TABLE имя_таблицы DISABLE ALL TRIGGERS;

Удаление
sql
DROP TRIGGER имя_триггера;

Информация о триггерах
sql
-- Словари данных
USER_TRIGGERS
ALL_TRIGGERS
DBA_TRIGGERS
USER_TRIGGER_COLS
ALL_TRIGGER_COLS

ОГРАНИЧЕНИЯ ТРИГГЕРОВ
Не могут выполнять транзакционные операторы (COMMIT, ROLLBACK, SAVEPOINT), за исключением автономных транзакций
Не могут изменять мутирующую таблицу (для строчных триггеров)
Не могут содержать слишком сложную логику (риск снижения производительности)
Не могут вызывать хранимые процедуры, которые нарушают указанные выше ограничения
Размер тела триггера ограничен (до 32К в некоторых версиях)

АВТОНОМНЫЕ ТРАНЗАКЦИИ В ТРИГГЕРАХ
Позволяют выполнять отдельные транзакции внутри триггера (например, для логирования ошибок) с помощью директивы:

sql
PRAGMA AUTONOMOUS_TRANSACTION;
Автономная транзакция может выполнять COMMIT или ROLLBACK независимо от основной транзакции.

ПРЕИМУЩЕСТВА ТРИГГЕРОВ
Централизованное управление бизнес-логикой на уровне БД
Гарантированное выполнение при любых изменениях данных
Прозрачность для приложений
Обеспечение сложных ограничений целостности
Аудит без изменения кода приложений

НЕДОСТАТКИ ТРИГГЕРОВ
Сложность отладки и сопровождения
Скрытая логика (неочевидна для разработчиков приложений)
Потенциальное снижение производительности
Риск каскадного срабатывания и бесконечных циклов
Ограничения мутирующих таблиц


РЕКОМЕНДАЦИИ ПО ИСПОЛЬЗОВАНИЮ
Использовать триггеры только для логики, которая должна выполняться при любых изменениях
Избегать сложной бизнес-логики в триггерах
Тщательно документировать все триггеры
Минимизировать количество триггеров на таблицу
Использовать BEFORE-триггеры для валидации данных
Использовать AFTER-триггеры для аудита и каскадных операций
Учитывать порядок выполнения при наличии нескольких триггеров
Тестировать производительность при массовых операциях
