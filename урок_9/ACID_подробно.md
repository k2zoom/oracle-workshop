ACID — ТРЕБОВАНИЯ К ТРАНЗАКЦИОННОЙ СИСТЕМЕ
Что такое ACID?
ACID — это аббревиатура, описывающая требования к надежности транзакционной системы (например, реляционной базы данных):

Буква	            Английский	        Русский                             В Oracle обеспечивается
A	Atomicity	    Атомарность	        Всё или ничего                      COMMIT / ROLLBACK, SAVEPOINT
C	Consistency	    Согласованность 	Данные всегда целостны              Constraints, триггеры
I	Isolation	    Изоляция	        Транзакции не мешают друг другу     Уровни изоляции, блокировки (FOR UPDATE)
D	Durability	    Долговечность	    После COMMIT данные не пропадут     Redo Logs, Archive Logs, Checkpoints

1. ATOMICITY (Атомарность)
Определение:
Транзакция выполняется целиком или не выполняется вообще. Нет промежуточного состояния.
Принцип "Всё или ничего":
Если транзакция выполняется успешно — все изменения сохраняются (COMMIT)
Если происходит ошибка — все изменения отменяются (ROLLBACK)


2. CONSISTENCY (Согласованность)
Определение:
Транзакция переводит базу данных из одного целостного состояния в другое. Все ограничения (constraints) должны соблюдаться.
Гарантии:
Нарушения ограничений вызывают ROLLBACK
Данные всегда соответствуют правилам (CHECK, UNIQUE, FOREIGN KEY)
Никакие "поломанные" данные не могут быть сохранены

Что обеспечивает согласованность:
Механизм	        Назначение
PRIMARY KEY	        Уникальность и идентификация
FOREIGN KEY	        Ссылочная целостность
UNIQUE	            Отсутствие дубликатов
CHECK	            Бизнес-правила
NOT NULL	        Обязательные поля
Триггеры	        Сложные проверки


3. ISOLATION (Изоляция)
Определение:
Транзакции не должны влиять друг на друга. Параллельное выполнение транзакций дает тот же результат, что и последовательное.

Проблемы параллельного доступа:
Проблема	            Описание	                                        Аналогия
Dirty Read	            Чтение незафиксированных данных другой транзакции	Увидеть зарплату до того, как ее утвердили
Non-repeatable Read	    При повторном чтении данные изменились	            Спросили цену, подумали, а она уже другая
Phantom Read	        Появились новые строки, которых не было	            Ищем столики, бронируем, а их уже заняли

Уровни изоляции (ANSI SQL):
Уровень	            Dirty Read	        Non-repeatable Read	        Phantom Read
READ UNCOMMITTED	Возможно	        Возможно	                Возможно
READ COMMITTED 	    Невозможно	        Возможно	                Возможно
REPEATABLE READ	    Невозможно	        Невозможно	                Возможно
SERIALIZABLE	    Невозможно	        Невозможно	                Невозможно
READ COMMITTED - Уровень изоляции в оракле по умолчанию


4. DURABILITY (Долговечность)
Определение:
После того как транзакция зафиксирована (COMMIT), данные сохраняются навсегда, даже при сбое системы, отключении питания или перезагрузке.

COMMIT
    ↓
Запись в Redo Log Buffer
    ↓
Запись в Redo Log File (на диск)
    ↓
Подтверждение пользователю
    ↓
(позже) Запись в Data Files

Механизмы обеспечения долговечности:

Механизм	            Назначение
Redo Logs	            Журнал повторного выполнения (все изменения)
Undo Logs	            Журнал отката (для ROLLBACK и чтения)
Checkpoint	            Синхронизация с data files
Archive Logs	        Архивированные redo logs для восстановления
